Hormuz Dock — API

Drive your deployments programmatically. Built to be friendly for AI agents (Claude & co.).

Everything the web panel does for Docker projects and static sites is available over a small REST API. Authenticate with a personal API key and you can list, deploy, restart, read logs, edit env, and more — no browser needed.

← Back to the panel

1. Get an API key

  1. Open the Hormuz panel → click API Keys in the top bar.
  2. Click + New key, give it a name (e.g. claude-deployer), and copy the key.
  3. The key is shown once — store it safely. It looks like hormuz_XXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX.

A key inherits your role (regular user or admin) and appears in the audit log tagged “via API key”. Revoke it anytime from the same screen.

Network access. The API lives behind the same gate as the panel. If your Hormuz is VPN-restricted (HORMUZ_PANEL_ALLOW_CIDRS), API calls must also come from the VPN. Public reverse-proxy routes (/_<slug>) and static sites (/_static_/) are unaffected.

2. Authenticate

Send the key as a bearer token (or an X-API-Key header). Confirm it works with /api/me:

# Base URL = wherever your panel lives, e.g. https://hormuz.qiscus.io
curl https://hormuz.qiscus.io/api/me \
  -H "Authorization: Bearer hormuz_XXXXXXXXXXXXXXXXXXXXXXXX"

# → {"id":2,"username":"you","role":"user","authVia":"apikey"}

3. Security model — what a key can & cannot do

API keys are deliberately scoped to the operational plane. They can run and manage projects, but they are barred from the identity & secret plane — so a leaked key can never escalate privileges, mint more keys, or exfiltrate global secrets. Those actions require an interactive browser login.

✓ Allowed with a key
  • List / create / delete projects & static sites
  • Start · stop · restart · rebuild · redeploy
  • Git pull, switch branch
  • Read logs, stats, container status
  • Read/write env, compose override, routes, files
  • System & storage info
✕ Blocked (403) — session only
  • User & role management (/api/users)
  • API-key management (/api/api-keys)
  • Global Secret Env (/api/secure-env)
  • Image prune (/api/system/prune)
  • SSH deploy key (/api/ssh-key)
  • The interactive shell (terminal WebSocket)
Tip for secrets. Don’t paste raw secrets into a project’s .env over the API. Instead reference the admin-managed Global Secret Env with @secure:<scope>/<KEY>; Hormuz resolves it only at compose up. The secret values themselves are never exposed to API keys.

4. Endpoint reference

All endpoints are under /api and return JSON. Mutations are recorded in the audit log with your identity.

GET/api/system/storagePer-container disk footprint + host disk usage

5. Recipes

Deploy the latest code for a project

# find the project id
curl -s $BASE/api/projects -H "Authorization: Bearer $KEY" \
  | jq -r '.[] | "\(.id)\t\(.name)\t\(.status)"'

# pull + rebuild + recreate
curl -X POST $BASE/api/projects/<id>/redeploy \
  -H "Authorization: Bearer $KEY"

Read the last 200 log lines

curl -s "$BASE/api/projects/<id>/logs?tail=200" \
  -H "Authorization: Bearer $KEY" | jq -r .logs

Set an env var (referencing a global secret)

curl -X PUT $BASE/api/projects/<id>/env \
  -H "Authorization: Bearer $KEY" -H "Content-Type: application/json" \
  -d '{"pairs":[{"key":"DATABASE_URL","value":"@secure:qismo/DATABASE_URL"},
                {"key":"LOG_LEVEL","value":"info"}]}'

Publish a static site from git

curl -X POST $BASE/api/static-sites \
  -H "Authorization: Bearer $KEY" -H "Content-Type: application/json" \
  -d '{"name":"Docs","source":"git","gitUrl":"https://github.com/acme/docs.git"}'
Errors. Non-2xx responses return {"error":"…"}. 401 = missing/invalid/revoked key · 403 = key hit a session-only endpoint (or wrong role/network) · 404 = not found · 409 = an operation is already running for that project.
